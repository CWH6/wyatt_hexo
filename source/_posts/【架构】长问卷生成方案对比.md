## 🎯 场景

一个 97 道题的问卷，题型包括单选、复选、矩阵、组合、MaxDiff、热力图等 20 种。需要生成一个 HTML 页面让用户答题。

在老设备（📱 内存小、CPU 慢）+ 网络差（🐌 2G/3G）的情况下，怎么生成这个页面才能让用户尽快看到第一道题、顺利答完全部题目？

下面是对比三种方案的思路和优缺点。👇



## 方案一：全部题目一次性生成到一个 HTML 📄

### 💡 思路

后端用 FreeMarker 模板把 97 道题的完整 HTML（DOM + 样式 + 脚本）全部塞进一个 HTML 文件，浏览器一次性下载并渲染。

```
后端渲染：97 道题 → 一个超大的 HTML 文件 → 浏览器下载后一次性显示
```

### ✅ 优点

- 实现最简单，不写额外 JS
- 所有题目一次性加载，后续切题瞬间完成，因为题都在 DOM 里
- 不需要接口、不需要 Ajax

### ❌ 缺点

- HTML 文件巨大（97 道题可能 1MB+ 💥），下载慢
- 浏览器要一次性解析全部 DOM，内存占用高，老设备可能直接卡死 😵
- 用户可能只答前几题就跑了，后面 90 道题的 HTML 白下载了 💸
- 首屏（第一道题出现）很慢 ⏳

### 🏠 适用场景

题目少（< 10 道），题型简单，网络好的 PC 端。

---

## 方案二："半骨架 + renderCache" 🦴

### 💡 思路

第一道题完整渲染在 HTML 里，其余题把 HTML 字符串存进一个 JS 变量 `__renderCache`，切换题目时从变量里取出对应字符串塞进 DOM。

```
后端渲染：首题直出 + 其余题 HTML 字符串 → 一个中大型 HTML 文件
前端切题：从 __renderCache 变量里取 HTML → 塞进 DOM
```

### ✅ 优点

- 首题直出，用户能比较快看到第一道题 👀
- 后续切题不依赖接口，从本地变量读，切换也很快 ⚡
- 比方案一的首屏 HTML 稍小一些（差不太多）

### ❌ 缺点

- `__renderCache` 变量仍然很大（几百 KB 的字符串），主 HTML 体积没显著下降 📦
- 浏览器仍需解析整坨大字符串，老设备内存压力和解析时间仍然很大 🥵
- 只有首题有 DOM，其他题是字符串，但终归都在主文件里，下载量没减少
- 属于"拆了一点，但没拆到位" 😐

### 🏠 适用场景

题目中等数量（10-30 道），希望有一定优化、但不想引入接口的情况。

---

## 方案三：单题按需异步加载 ⚡

### 💡 思路

主 HTML 只包含每道题的外层壳子（`<div class="question">` + 几个属性），没有任何题目的真实内容。用户打开页面时，第一道题通过接口异步加载；点击下一题时，再发一个接口请求加载下一道题的内容。

```
主 HTML：只包含空壳（约 50KB）📦
    ↓ 页面打开，异步加载第 1 题
接口 1：返回第 1 题的完整 HTML（含样式 + 脚本）→ 注入 DOM ✅
    ↓ 点下一题，异步加载第 2 题
接口 2：返回第 2 题的完整 HTML → 注入 DOM ✅
    ↓ ...
```

### ✅ 优点

- **主 HTML 极小**（~50KB），下载快、解析快 🚀
- **老设备最友好**：不用一次性解析全部题的 DOM，内存压力大幅下降 🧊
- **按需加载**：用户只加载自己真正看到的题目，看了 3 道题就跑了，只加载了 3 道 🎯
- **网络差也能用**：虽然每道题需要请求接口，但每次只传一道题（几 KB），比一次性传几百 KB 快 📶
- 可以加 loading 动画，用户体验不差 🌀

### ❌ 缺点

- 需要额外写一个后端接口 🔧
- 每次切题多一个网络请求（但题目 HTML 经过 gzip 压缩后只有几 KB）
- 断网时无法切题（但可以配合本地缓存策略）📴

### 🏠 适用场景

题目多（> 30 道），老设备、弱网环境，追求首屏速度。**🏆 这是长问卷的最佳方案。**

---

## 📈 三种方案对比

| 对比维度 | 方案一：全量 HTML | 方案二：半骨架 | 方案三：按需异步加载 |
|---|---|---|---|
| 主 HTML 大小 | 很大（1MB+ 💥） | 较大（600KB+ 📦） | **很小（~50KB ✅）** |
| 首题出现速度 | 🐌 慢 | 🚶 中等 | **🚀 快** |
| 下载总量 | 大，一次性全部 | 较大，一次性全部 | 小，按需分批 |
| 内存压力 | 🔴 高 | 🟡 较高 | **🟢 低** |
| 老设备表现 | 😵 差 | 😐 一般 | **😊 好** |
| 弱网表现 | ❌ 差 | ⚠️ 较差 | **✅ 好** |
| 后续切题速度 | ⚡ 快 | ⚡ 快 | 🌐 需等接口（但很快） |
| 实现复杂度 | ⭐ 极简 | ⭐⭐ 简单 | ⭐⭐⭐ 中等 |

## 🏆 为什么方案三最适合长问卷

把问卷想象成一本书 📖：

- **📄 方案一**：把整本书印在一张巨大的纸上，递给你。纸太重，你的桌子（内存）放不下。
- **📋 方案二**：把第一页印在纸上，剩下的页折起来夹在里面。纸还是很厚很重。
- **📱 方案三**：只给你一个空壳封面，你想看哪一页，就单独拿哪一页过来。书很轻，桌子很干净。

对于老设备（内存小 📱）+ 网络差（2G/3G 📶）的场景：

1. 主 HTML 只有 50KB，即使 2G 网络也能在 1-2 秒内下载完成 ⚡
2. 浏览器只需要解析 50KB 的空壳 HTML + 第一道题的内容，内存占用极低 🧊
3. 每道题单独加载时，题目 HTML 经过 gzip 压缩通常只有几 KB，接口响应很快 📦
4. 配合缓存策略 + gzip，二次加载几乎零传输 🔄

> **🎯 结论：方案三是长问卷在老设备 + 弱网环境下的最优解。**
