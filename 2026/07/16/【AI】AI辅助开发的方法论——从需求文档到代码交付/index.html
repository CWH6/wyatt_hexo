【AI】AI辅助开发的方法论——从需求文档到代码交付

发表于2026-07-16|更新于2026-07-16|AI设计
|浏览量:

用好 AI 写代码，拼的不是提示词技巧，而是你有没有把”需求想清楚”。本文整理一套可复用的 AI 辅助开发流程——从需求文档、功能拆分，到提示词沉淀，形成你自己的开发流水线。

背景

用 AI（Trae / Cursor / Claude）写代码有一段时间了，踩过不少坑，也摸索出了一些规律。最大的体会是：AI 能写代码，但它不能替你思考。

如果你上来就说”帮我写一个电商系统”，AI 也能给你产出，但大概率是一坨不可维护的代码。原因很简单——它不知道你的业务上下文、技术约束、团队规范。

所以我慢慢形成了一套工作方式：在让 AI 写代码之前，先写清楚”要做什么”。这篇文章就是这套方法的总结。

核心理念

需求文档（人写）→ 功能拆分（人+AI）→ 技术设计（人+AI）→ 编码实现（AI为主）→ 测试与问题追踪（人+AI）→ 审查沉淀（人为主）

六个阶段，越往前人的参与度越高，越往后 AI 的参与度越高。关键是把”人该想清楚的事情”和”AI 擅长做的事情”分开。

文档结构设计

我推荐在项目根目录下建立 docs/ 目录，按以下结构组织：

docs/
├── 01-需求文档/
│   ├── README.md              # 需求总览，一句话说清这个项目要做什么
│   ├── 业务背景与目标.md       # 为什么要做？解决什么问题？成功标准是什么？
│   ├── 用户故事.md             # 谁？要做什么？为什么？
│   └── 非功能需求.md           # 性能指标、安全要求、兼容性约束
│
├── 02-功能模块/
│   ├── 模块A-用户管理/
│   │   ├── 功能说明.md         # 这个模块负责什么，有哪些页面/操作
│   │   ├── 接口设计.md         # 前后端契约：API 路径、请求/响应结构
│   │   ├── 后端/
│   │   │   ├── 数据模型.md     # 表结构、字段说明、关联关系
│   │   │   └── 业务逻辑.md     # 核心流程、状态机、校验规则
│   │   └── 前端/
│   │       ├── 页面设计.md     # 路由、页面结构、组件树
│   │       └── 组件设计.md     # 关键组件的 props/events/slots
│   └── 模块B-订单系统/
│       ├── 功能说明.md
│       ├── 接口设计.md
│       ├── 后端/
│       │   ├── 数据模型.md
│       │   └── 业务逻辑.md
│       └── 前端/
│           ├── 页面设计.md
│           └── 组件设计.md
│
├── 03-技术设计/
│   ├── 架构图.md               # 系统架构、模块关系、数据流
│   ├── 技术选型.md             # 用什么框架/中间件/数据库，为什么选它
│   └── 部署方案.md             # 环境、CI/CD、监控
│
├── 04-AI提示词/
│   ├── 需求分析提示词.md        # 让 AI 帮忙梳理需求时用的 prompt
│   ├── 后端编码提示词.md        # 生成后端代码（Spring Boot）时用的 prompt
│   ├── 前端编码提示词.md        # 生成前端页面（Vue/React）时用的 prompt
│   ├── 问题诊断提示词.md        # 排查 Bug / 安全漏洞时用的 prompt
│   └── 代码审查提示词.md        # 让 AI review 代码时用的 prompt
│
├── 05-开发日志/
│   └── 迭代记录.md              # 每次 AI 对话的关键决策、踩坑记录
│
└── 06-问题追踪/
    ├── 20260716_登录页XSS漏洞.md   # 日期_问题描述，一个文件一个 Bug
    ├── 20260717_订单导出超时.md
    └── 20260718_移动端图片加载失败.md

为什么这样设计？

编号前缀（01-06） 保证文件夹按流程排序，新加入项目的人也能按顺序阅读。

每个功能模块独立一个文件夹，而不是把所有接口设计放在一个大文件里。这样做的好处是：

  • 给 AI 看上下文时，只需要把当前模块的文档丢给它，不会超 token
  • 模块之间解耦，改 A 模块不影响 B 模块的文档

模块内按前后端拆分，以 接口设计.md 作为前后端的契约文件——后端按接口实现，前端按接口调用。这样前后端可以并行开发，只要契约不变，两边互不阻塞。

04-AI提示词/ 是一个容易被忽略但非常重要的目录。好的 prompt 需要反复调试，调试好了就沉淀下来。下次做类似需求时直接复用，效率翻倍。

06-问题追踪/ 采用 日期_问题描述.md 的命名方式，一个文件一个 Bug。这样做的好处是：

  • 按日期自然排序，一眼看出项目的时间线
  • 文件名即摘要，搜索时直接 grep 文件名即可定位
  • 每个文件记录完整的排查过程（现象 → 分析 → 修复 → 验证），下次同类问题直接翻出来参考
  • 丢给 AI 时只需喂一个文件，上下文精炼，诊断更准

团队协作：文档放哪里？

前后端分离项目通常是两个独立仓库（project-backend + project-frontend），文档放哪个仓库都不对——另一边看不到，两边各放一份又必然不同步。

结论很简单：前后端没拆就用 Monorepo，已经拆了就加一个 docs 仓库，三仓库是终点方案。

场景 方案 仓库数
项目刚开始，还没拆 Monorepo，backend/ + frontend/ + docs/ 放一起 1 个
前后端已经独立仓库 加一个独立 project-docs 仓库 3 个

为什么不用 Git Submodule 把 docs 嵌入到两个仓库里？多人协作时 submodule 版本容易不一致、忘记 git submodule update，徒增心智负担。独立 docs 仓库虽然多了一个，但它只是个纯 markdown 仓库，不需要 CI/CD、不需要部署、不存在维护成本。

三个仓库放在同一个父目录下：

~/projects/my-app/
├── my-app-backend/     # 后端代码
├── my-app-frontend/    # 前端代码
└── my-app-docs/        # 文档（结构见上文）

这样做的好处：

  • 单一真相源：接口变更在这里改，前后端同步看到
  • 权限分离：文档仓库可以单独设置谁有写权限（比如只有 Tech Lead 能改接口设计）
  • AI 友好：拉一个新对话窗口，把三个仓库的父目录告诉 AI，它就能读取全部上下文
  • Git 历史即变更记录：谁在什么时候改了哪个接口，git log 一目了然

日常怎么用 AI 引用文档？

不要打开单个仓库，打开父目录。 这是最关键的一步。

Trae / Cursor 这类 IDE 里，用”打开文件夹”打开 ~/projects/my-app/（父目录），而不是 my-app-backend/。这样 AI 能同时看到三个子目录的文件：

~/projects/my-app/        ← 在 IDE 里打开这一层
├── my-app-backend/
├── my-app-frontend/
└── my-app-docs/

写后端代码时，提示词里直接引用文档路径：

请根据 my-app-docs/02-功能模块/用户管理/接口设计.md 和 
my-app-docs/02-功能模块/用户管理/后端/数据模型.md，
帮我实现 UserController 的 CRUD 接口。

AI 会自己去对应路径下读文件，不需要你手动把文档内容贴进来。

日常惯例：

场景 怎么做
需求阶段 在 docs 仓库直接写文档，AI 帮你审阅
写后端 打开父目录，引用 my-app-docs/ 下的接口设计和后端文档
写前端 打开父目录，引用 my-app-docs/ 下的接口设计和前端文档
改接口 先改 docs 里的接口设计，commit push，前后端再各自跟着改代码
提 Bug 在 docs 的 06-问题追踪/ 写问题文件，AI 诊断后回到对应代码仓库修复

核心原则：文档先行，代码后行。 永远先改文档再改代码，保证文档始终是代码的”最新说明书”。

六阶段实操

第一阶段：需求文档（人写，AI 辅助梳理）

这是最关键的一步。你不一定要写得多正式，但至少要回答清楚三个问题：

  1. 要解决什么问题？ —— 写给谁用？在什么场景下用？
  2. 成功的标准是什么？ —— 做到什么程度算”做好了”？
  3. 有什么约束条件？ —— 技术栈限制、时间限制、合规要求

写完初稿后，可以把需求文档丢给 AI，让它帮你：

  • 找出模糊的地方（”这个用户故事缺少验收条件”）
  • 补充边界 case（”如果用户上传的文件超过 10MB 怎么办？”）
  • 发现遗漏的功能点（”你们需要考虑密码找回流程吗？”）

示例提示词（沉淀到 04-AI提示词/需求分析提示词.md）：

你是一个资深产品经理。请阅读以下需求文档，帮我做三件事：
1. 指出表述模糊、容易引起歧义的地方
2. 列出可能遗漏的边界情况和异常流程
3. 补充你可能想到的、但文档没提到的功能点

第二阶段：功能拆分（人主导，AI 辅助细化）

有了需求文档，就可以按功能模块拆分了。拆分的粒度建议是：一个模块 = 一个独立可交付的功能单元。

比如一个”问卷调查系统”可以拆成：

模块 核心功能
用户管理 注册、登录、权限控制
问卷编辑 创建问卷、添加题目、题型配置
问卷发布 生成链接、二维码、渠道管理
数据收集 答卷存储、进度保存、防重复提交
数据分析 统计图表、交叉分析、数据导出

每个模块对应一个文件夹，先写共享的契约文件，再分别补充前后端实现文档：

  • 功能说明.md：这个模块做什么，有哪些页面/操作，交互流程是怎样的
  • 接口设计.md：前后端契约，定义 API 路径、Method、请求参数、响应格式。这份文档前后端共用，是联调的基准。
  • 后端/：数据模型 + 业务逻辑（核心流程、状态机、校验规则）
  • 前端/：页面设计（路由、组件树）+ 组件设计（关键组件的 props/events）

第三阶段：技术设计（人+AI 协作）

这一步决定系统的骨架。核心决策包括：

  • 架构模式：单体还是微服务？分层架构还是 DDD？
  • 技术选型：为什么选这个框架 / 中间件？（别拍脑袋，写清理由）
  • 数据流：请求从前端到后端再到数据库，经过哪些环节？
  • 部署方案：开发/测试/生产环境的配置差异

技术选型部分建议单独写一份文档，记录每个决策的”为什么”。三个月后回来看，你会感谢现在的自己。

第四阶段：编码实现（AI 为主，人为辅）

文档到位了，编码反而是最快的一步。前后端分离项目按以下顺序推进：

Step 1：先定契约，后端先行

接口设计.md 就是前后端契约。先把接口文档丢给 AI 生成后端代码：

  1. 喂入模块的 功能说明.md + 接口设计.md + 后端/数据模型.md + 后端/业务逻辑.md
  2. 让它先生成数据模型（Entity / DDL），你确认表结构没问题
  3. 再让它生成 API 层（Controller / Service / Mapper）
  4. 用 Postman / Apifox 快速验证接口可用
  5. 让 AI 补充单元测试

后端编码提示词（沉淀到 04-AI提示词/后端编码提示词.md）：

请根据以下功能模块文档，帮我实现 XXX 模块的后端代码。
技术栈：Spring Boot + MyBatis-Plus + MySQL

要求：
1. 先读 {功能说明.md}，理解业务逻辑
2. 参考 {接口设计.md} 生成 Controller 和 Service
3. 根据 {数据模型.md} 生成 Entity 和 Mapper
4. 统一返回格式为 Result<T>，异常用全局异常处理器
5. 每个接口写上清晰的注释

Step 2：接口就绪后，前端跟进

后端接口跑通后（至少 Swagger 能访问），开始搞前端：

  1. 喂入 功能说明.md + 接口设计.md + 前端/页面设计.md + 前端/组件设计.md
  2. 让它生成页面路由和基础组件骨架
  3. 对接后端接口，处理 loading / empty / error 三种状态
  4. 逐个组件细化交互逻辑

前端编码提示词（沉淀到 04-AI提示词/前端编码提示词.md）：

请根据以下文档，帮我实现 XXX 模块的前端页面。
技术栈：Vue 3 + TypeScript + Element Plus

要求：
1. 先读 {页面设计.md}，了解路由和页面结构
2. 参考 {接口设计.md} 对接后端 API，统一用 request.ts 封装
3. 根据 {组件设计.md} 实现关键组件的 props/events
4. 每个接口调用都要处理 loading、empty、error 三种状态
5. 移动端适配，使用 rem / flex 布局

Step 3：联调

前后端都跑通后，联调一般只需要微调——因为两边的实现都基于同一份 接口设计.md，出问题的概率大幅降低。

注意：一次只喂一个模块的文档，避免上下文过长导致 AI 顾此失彼。

第五阶段：测试与问题追踪（人+AI 协作）

编码完成不代表能交付。这个阶段的思路是：当日问题当日记录，丢给 AI 诊断修复，沉淀为可检索的知识库。

操作流程：

  1. 测试发现问题后，在 06-问题追踪/ 下创建文件，命名格式 YYYYMMDD_问题简述.md
  2. 文件模板建议包含四个板块：现象 → 分析 → 修复 → 验证
  3. 把文件内容喂给 AI，让它帮你诊断根因并给出修复方案
  4. 验证通过后，把 AI 的诊断过程和最终方案填回文件，形成闭环

文件模板：

# 20260716_登录页XSS漏洞

## 现象
- 在登录页输入框输入 `<script>alert(1)</script>` 后提交，页面弹窗
- 浏览器：Chrome 120，复现率 100%

## 分析
- （AI 诊断结果填这里）根因：前端未对用户输入做 HTML 转义，后端也没做校验
- 影响范围：仅登录页的用户名输入框

## 修复
- 前端：`v-html` 改为 `{{ }}` 模板插值，或使用 DOMPurify 清洗
- 后端：添加 `@XssFilter` 注解统一过滤
- 涉及文件：`Login.vue` L45、`UserController.java` L32

## 验证
- [x] 输入 XSS payload 不再弹窗
- [x] 正常用户名仍可登录
- [x] 回归测试通过

为什么这个模块有价值？

三个月后你遇到一个”某个输入框又弹窗了”的问题，直接在 06-问题追踪/ 下搜索 XSS，之前的所有排查记录都在。不用重新分析，直接复用修复方案。

问题诊断提示词（沉淀到 04-AI提示词/问题诊断提示词.md）：

你是一个资深全栈工程师。我遇到了以下问题，请帮我诊断并给出修复方案。

技术栈：Spring Boot + Vue 3 + MySQL

请按以下结构输出：
1. 根因分析：为什么会发生这个问题？
2. 影响范围：会影响哪些模块/用户？
3. 修复方案：具体改哪些文件、怎么改（给出代码 diff）
4. 预防措施：如何避免同类问题再次出现？

第六阶段：审查沉淀（人为主）

代码写完了不等于结束了，还有三件事：

  1. 代码审查：让 AI review 自己的代码（换个对话窗口，不要用同一个 session）
  2. 提示词沉淀：这次调试好的 prompt 写进 04-AI提示词/，下次直接用
  3. 踩坑记录：AI 这次在哪个环节容易出错？你是怎么纠正的？记到 05-开发日志/迭代记录.md

示例提示词（沉淀到 04-AI提示词/代码审查提示词.md）：

请 review 以下代码，从以下几个方面给出意见：
1. 逻辑是否正确？有没有边界条件没考虑？
2. 是否存在安全漏洞（SQL注入、XSS、越权）？
3. 命名是否清晰？注释是否到位？
4. 有没有可以优化的性能点（N+1查询、重复计算）？
5. 错误处理是否完善？

总结

这套方法的核心就一句话：用文档把需求锁死，让 AI 在边界内发挥。

阶段 产出物 人的角色 AI 的角色
需求文档 01-需求文档/ 写初稿 补充遗漏、找模糊点
功能拆分 02-功能模块/ 定拆分粒度 细化接口和模型
技术设计 03-技术设计/ 做决策 给建议、画架构图
编码实现 代码 确认方向 主力输出
测试与问题追踪 06-问题追踪/ 发现问题、验证修复 诊断根因、给出修复方案
审查沉淀 04-AI提示词/ 05-开发日志/ 审核 + 记录 辅助审查

刚开始可能会觉得”写文档比写代码还慢”，但当你做了两三个项目、积累了自己的提示词库之后，就会发现复利效应——下一个项目只需要改文档内容，流程直接复用。
